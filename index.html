单独看，三篇像三个不相干的功能。叠在一起，它们是一张坐标系：skills 给深度、loop 给时间、workflow 给广度。三条轴交汇处，就是这套工具想去的终点。

那个终点，是一个你只需定义目标、它就能自己排期、自己判断、自己分工的自主系统。下面每一章都用同一个节奏：它是什么，核心怎么拆，关键洞察在哪，以及怎么上手。

01
Getting Started with Loops

Loops：让 Claude 自己转起来

Loop 就是让 agent 重复一轮又一轮地工作，直到满足某个停止条件。重点不是写一句更聪明的 prompt，而是设计一套让它能自主迭代的系统。

四种 loop，按"谁触发、何时停"分

类型触发停止条件适合命令
逐轮用户 promptClaude 判断做完，或需要更多输入短的、一次性任务默认模式
目标手动实时 prompt达成目标，或到最大轮数有可衡量退出标准的任务/goal
定时指定的时间间隔用户取消，或工作完成重复性工作、对接外部系统/loop /schedule
主动事件或计划，无人实时参与单任务达标；例程持续到被关闭定义清晰的重复工作（bug 分诊、迁移、依赖更新）多种组合

三个要记住的点

/goal

用目标消灭"够好了吗"

一个评估模型检查是否真达标，避免它半途收工。例：/goal 把首页 Lighthouse 跑到 90+，试 5 次后停。

/loop vs /schedule

本地 vs 云端

/loop 在本地按间隔重跑；/schedule 把任务搬到云端持续执行，不再依赖你的电脑开着。

self-verify

把验证写成 skill

把手动检查步骤编码成 skill，让 Claude 端到端自检（起服务，点浏览器，看 console，跑性能审计），loop 才敢放手。

主动 loop 长什么样 一个真实例子把上面几块拼在了一起：「/schedule 每小时：检查 #project-feedback 频道的 bug 报告。/goal：本轮找到的每条报告不分诊完、不处理完、不回复完，就不许停。」 这就是无人值守自动化的雏形。

省 token：按复杂度选模型（小任务给更快的模型，判断题才动用最强模型）；把成功和停止标准写精确；先在小数据集上试点再放大；确定性的活用脚本做、别让它推理；例程间隔匹配真实的变化频率；用 /usage /workflows 盯消耗。保质量：代码库保持整洁，让它跟随既有模式；把质量标准写进 skill 让它自检；派第二个 agent 以全新视角做审查。

一句话 从最简单、够用的方式起步。观察它卡在哪、在哪越界、哪里被搞复杂了，再据此迭代，别一上来就堆复杂 loop。
02
Lessons from Building Claude Code · How We Use Skills

Skills：把专长打包给 Claude

Skill 是一个文件夹，装着指令、脚本和资源，让 agent 能自己发现并调用。常见误解是"它不就是几个 markdown 吗"：其实它是能包含脚本、素材、动态 hook 的结构化目录。

九类 skill

Anthropic 内部把数百个 skill 归成九类，几乎覆盖所有工程场景。

库 / API 参考 产品验证 数据获取与分析 业务流程自动化 脚手架 / 模板 代码质量与审查 CI/CD 与部署 Runbooks 基础设施运维

写好 skill 的九条心法点开看

不要重复 Claude 已知的东西。要给真正新的洞察，把它推出惯性思维。比如前端设计 skill 就专门教它避开 Inter 字体、紫色渐变这类烂大街的默认套路。
价值最高的内容是记录那些反复失败的点：数据库怪癖（比如某张表是 append-only）、跨系统字段命名不一致、只在生产环境才出现的行为。
用文件夹分层（references / scripts / examples），按当下情境把 Claude 指向具体文件，而不是把所有东西塞进一个大文档撑爆上下文。
给足必要信息，但保留灵活性。让 agent 能按情境调整做法，而不是被强制走一套死板步骤。
有些 skill 需要用户配置（存在 config.json）。信息缺失时，用 AskUserQuestion 工具做结构化的多选提问，而不是瞎猜。
skill 的描述决定它何时被触发。要塞进激活关键词（比如 "babysit"），提升 Claude 识别该用它的准确率。
用 ${CLAUDE_PLUGIN_DATA} 环境变量存持久数据：append-only 的日志记录做过什么，或用 JSON 文件维护状态。
提供现成的库和辅助函数，让 Claude 专注于组合，而不是每次重造样板。复杂分析和自动化会因此大幅提速。
只在 skill 被调用时才激活的 hook。例如 /careful（拦截破坏性命令）、/freeze（限制只能改指定目录）。
一个小小的自我实践 你正在看的这版页面，就是照第 01 条改的：手搓的第一版用了暖米色底 + 陶土色 + 衬线大标题，正是官方文档点名的"AI 默认套路"。这版换成了暗色代码编辑器配色 + Space Grotesk 无衬线。设计感，往往来自"知道该避开什么"。

分发：小团队少仓库，直接 check 进 ./.claude/skills；要规模化就建内部插件市场，团队按需安装。一个 skill 可以先在 sandbox 里有机生长，被广泛采用后再进市场。度量：用 PreToolUse hook 记录使用，找出热门的，也揪出"该触发却没触发"的欠触发。

03
A Harness for Every Task · Dynamic Workflows

Workflows：给每个任务配一套指挥系统

Dynamic workflow 是让 Claude 临场为特定任务写一套定制的多 agent 协作系统，而不是所有活儿都套用同一个静态框架。这套指挥框架叫 harness；默认那套是为编码优化的，但很多别的任务需要专门的协调策略。

它解决什么：单上下文的三种翻车

当 Claude 在一个上下文窗口里硬扛复杂任务时，会有三种典型失效。解法是 spawn 出一批各有隔离上下文、目标聚焦的 subagent。

Agentic Laziness

偷懒式收工

做了一部分就宣布完成。50 项的安全审查，做了 35 项就交差。

Self-Preferential Bias

自我偏袒

让它验证或评判自己的产出时，它会偏心自己的结论。

Goal Drift

目标漂移

多轮之后偏离原目标，约束在一次次 summary 里被悄悄丢掉。

六种编排范式

01

Classify-and-Act分类再行动

一个分类 agent 先判断任务类型，再路由到不同 agent 或行为。

02

Fan-Out-and-Synthesize扇出再综合

拆成并行小步各自独立跑，再综合。综合步是一道栅栏，会等齐所有 subagent。

03

Adversarial Verification对抗式验证

为每个产出再派验证 agent，按特定 rubric 去挑刺。

04

Generate-and-Filter生成再筛选

先生成一堆方案，按 rubric 过滤、去重，只留最高质量的少数。

05

Tournament锦标赛

多个 agent 用不同方法竞争同一任务，评委用成对比较选胜者。

06

Loop Until Done干到完成

开放式工作，反复 spawn 直到满足停止条件，而非固定跑几次。

技术上怎么跑 动态 workflow 会执行一个 JavaScript 文件，里面有专门的函数来 spawn 和协调 subagent（也能用 JSON、Math、Array 处理数据）。它还能决定每个 subagent 用哪个模型、让它们在隔离的 worktree 里跑、以及从上次的 checkpoint 断点续跑。

用武之地：大迁移与重构（Bun 把 Zig 重写成 Rust 就用了这套）、深度研究、深度事实核查、排序与排名（成对比较比绝对打分更可靠）、根因调查（多个独立假设防自我偏袒）、大规模分诊（用隔离模式限制处理不可信内容的 agent 触碰高权限操作），以及设计、命名这类靠品味的探索决策。

该用

复杂、高价值的任务；探索性工作；需要验证的场景；天然可并行的问题。代价是比单 agent 更费 token。

别用

常规编码通常不需要一整套多 agent 评审团。动手前先问自己一句：这真的需要更多算力吗？

动态 vs 静态：静态 workflow（用 Agent SDK 或 claude -p 写）能覆盖所有边界但偏通用；在 Opus 4.8 下，动态 workflow 能针对具体问题定制最优编排。上手：prompt 点名具体范式，搭配 /goal 和 /loop，给明确 token 预算（如"用 10k tokens"），在菜单里按 s 保存，存到 ~/.claude/workflows 或用 skill 分发，并把共享的当模板保持灵活。

三轴合体

拼在一起，才是完整那张图

三根支柱不是并列的功能，而是同一个自主系统的三个维度。它们各管一段，缺一段，系统就转不起来。

↕
Loops
时间轴

Claude 何时、干多久。让它持续、重复、无人值守地运转下去。

↓
Skills
深度轴

Claude 懂什么、会什么。把专长和踩坑经验沉淀成可复用的资产。

↔
Workflows
广度轴

Claude 如何分身。临场指挥多个隔离的自己，并行攻坚。

主动 loop（Proactive Loop） =
/schedule 定时 + /goal 目标 + skills 知识 + dynamic workflows 并行 + auto mode
这套合体形态，就是一个能自己排期、自己判断完成、自己调用专长、自己分派子任务的系统。

心智模型的转变：从"我问一句、它答一句"，走向"我定义一个能自运转的系统"。这正是三篇官方文档共同指向的方向。
